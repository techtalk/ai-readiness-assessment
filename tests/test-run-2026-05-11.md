## In-session changes applied

This run reflects two corrections made between the first pass and the
final pass:

### 1. Engagement map extended to cover L3→L4 and L4→L5 jumps

The first pass (only L0–L3 fixtures) surfaced a real gap: the
engagement map in step 6 of the skill had three rows, one per
discipline, but no row for the cross-cutting level-transition cases.

Two rows added to both `SKILL.md` and `commands/ai-readiness-assess.md`
(dual-surface sync) plus a short explanatory note:

| Weakest gap | Suggested engagement type |
|---|---|
| Specifications layer (L3→L4 jump) | Specification-first engagement — install a `specs/` layer, spec-conformance constraints in HARNESS.md, and an adversarial-review touchpoint at plan approval. |
| Sovereign / platform layer (L4→L5 jump) | Platform-engineering engagement — package the team's habitat as a published artefact (plugin, template, marketplace entry), install a governance audit cadence, decision archaeology (CHOICES.md or story records), fitness functions, and cost/model-routing discipline. |

The L3, L4, and L5 fixtures all exercise these new rows and pass A11
cleanly.

### 2. B3 reclassified as interactive-only

The first pass uniformly failed B3 across every fixture because
batch mode cannot ask the contract's 3–5 clarifying questions. This
was a test-methodology artefact, not a skill defect.

All six `expected.md` files updated:

> B3. **(Interactive runs only)** 3–5 clarifying questions in total.
> **N/A in batch test runs** that cannot accept user input — mark
> N/A, not FAIL. The contract still holds for interactive use.

Batch runs now produce 17 N/A instead of 6 FAIL + 11 N/A.

## What the run revealed (this time)

1. **The skill is robust across the full framework.** Every
   structural and semantic assertion passes on every fixture from L0
   to L5. The scoring heuristic, the reading-path lookup, the
   gap-anchored CTA pattern, and the engagement map (now extended)
   all work consistently.

2. **L5 framing is correct.** The most subtle test was whether the
   L5 fixture would prompt the skill to invent a non-existent
   Level 6 — it didn't. The L5 CTA explicitly says "The aim isn't a
   sixth level; it's compounding the practice you already have."

3. **The dual-surface sync convention paid for itself.** The
   engagement-map update touched two files; making the edit
   identically in both was straightforward because the convention is
   explicit in `HARNESS.md`. This is the kind of payoff a content
   invariant produces.

4. **No further skill-prose changes recommended from this run.**
   Both findings from the first pass are now closed.

## Open work

1. **An interactive run is still needed** to verify the B-tier
   behavioural assertions (scan-first sequencing, questions-one-at-
   a-time, the 3–5 question count). Batch mode cannot exercise
   these.

2. **The dual-surface sync constraint is not yet automated.** The
   `HARNESS.md` convention says command and skill must contain
   identical framework content; this run made that change manually
   in both files. A future TDAB test could diff specific sections of
   the two files and assert byte-equality (deterministic) or
   semantic equivalence (agent). Promote on the harness ladder when
   the team is ready.

3. **A real test runner would replace this manual process.** Each
   fixture has its `expected.md`; a script that invokes the skill
   against each fixture, parses the assessment, and emits the test
   report automatically would convert this from "TDAB with a human
   in the loop" into "agent-verified TDAB" — the next rung on the
   promotion ladder.

4. **The "scripts/ referenced but not implemented" note in the L5
   fixture** is a fidelity gap, not a habitat gap. If we want
   strict fidelity, add the script files (even stubs) to the L5
   fixture. Optional.
