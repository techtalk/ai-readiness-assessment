## What the run revealed

### 1. Skill prose is structurally and semantically sound at L0–L3

Every structural and semantic assertion passed across all four
fixtures without prose changes. The scoring heuristic, the
discipline-maturity table, the reading-map lookup, and the
gap-anchored CTA pattern all behave correctly across the lower half
of the framework.

### 2. The engagement map (SKILL.md, step 6) is incomplete for the L3→L4 jump

The L3 fixture's CTA had to invent a "specification-first engagement"
that isn't named in the canonical engagement map. The map only covers
the three disciplines as first-order labels; the cross-cutting L3→L4
move (making specs first-class) has no row.

**Proposed change** (defer to user — this is a real prose edit and
should be reviewed):

> `commands/ai-readiness-assess.md`, step 6 (Next steps / TechTalk
> CTA), add a fourth row to the engagement map:

> | Specifications layer (L3→L4 jump) | Specification-first engagement — guided work that produces a `specs/` layer, spec-conformance constraints in `HARNESS.md`, and an adversarial-review touchpoint at plan approval. |
>
> Dual-surface sync applies: both files must change together.

### 3. B3 is a real contract failure under batch mode

The skill's contract requires 3–5 clarifying questions, asked one at
a time. A batch test run cannot honour this. Three options:

1. **Accept the batch limitation**: mark B3 as "interactive-only" in
   `expected.md` and exclude it from automated runs. The contract
   stays as-is for interactive use.
2. **Add an explicit batch mode to the skill**: a flag that skips
   step 2 and records "questions skipped — batch mode" in the
   assessment file. Makes batch-mode runs first-class.
3. **Build a question-stub runner**: provide canned answers for the
   questions the skill asks. The least pragmatic option — defers all
   the interesting test logic to the canned answers.

Recommendation: option 1 for now. Re-evaluate when there's an
interactive test runner.

### 4. The dual-surface sync constraint is in scope but not yet tested

The five conventions in `HARNESS.md` include "Dual-surface sync"
(command and skill carry identical framework content). The test
fixtures exercise the skill's behaviour but do not verify that the
command produces the same behaviour as the skill. A future test:
invoke `/ai-readiness-assess` and the skill via natural-language
trigger against the same fixture, diff the outputs, assert they are
behaviourally equivalent.
